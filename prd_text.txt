Product Requirements Document (PRD): Port Intelligence System
Project Title: Smart Port Intelligence System (SPIS)
Objective: To  build  an  integrated  smart  port  intelligence  system  capable  of  forecasting  throughput,
detecting anomalies via chatbot queries, scheduling predictive maintenance using synthetic sensor data,
and optimizing berth assignment. The platform will support ground-level staff via a secure login system,
integrating facial recognition and traditional credentials, and be deployed using containerized services.
System Overview
Core Features: 1. Port Throughput Forecasting for Capacity Planning 2. Anomaly Detection + Chatbot
Escalation  System  3.  Smart  Predictive  Maintenance  Scheduling  (with  synthetic  dataset)  4.  Route
Optimization and Berth Scheduling (API-integrated, Phase II)
Supporting Components: - Secure Login with Facial Recognition + Password - Text-to-SQL Query Interface
(via chatbot for dynamic querying) - Docker-based microservice deployment on a free-tier cloud platform
Feature 1: Port Throughput Forecasting
Model Type: LSTM (Long Short-Term Memory)
Input: - Historical throughput volume (TEUs, cargo units) - Port congestion level - Weather severity -
Shipping cost
Output: - Forecasted cargo volume (daily/weekly/monthly)
Evaluation Metrics: - Mean Absolute Error (MAE) - Root Mean Squared Error (RMSE) - R-squared (R²) score
Fine-Tuning  Strategy: -  Hyperparameter  tuning  (hidden  layers,  window  size,  dropout  rate)  -  Rolling
validation using walk-forward approach
Feature 2: Anomaly Detection + Chatbot Escalation
Tasks: a) Anomaly Detection - Continuously monitors structured port operations data for abnormalities. -
Uses autoencoders to identify anomalies in operational parameters (e.g. equipment usage, traffic flow,
delays).
b) Chatbot Issue Escalation System - Accepts natural language issue descriptions from port staff. - Uses
NLP + classification to determine: - Known issue ➔ provide resolution - Unknown/complex issue ➔ escalate

Model Type: - Autoencoder (Anomaly Detection) - BERT or DistilBERT (Intent Classification)
Evaluation  Metrics: -  Anomaly  Detection:  Reconstruction  Error  Threshold,  AUC-ROC  -  Classification:
Accuracy, Precision, Recall, F1-Score
Fine-Tuning Strategy: - Threshold calibration for autoencoder loss - Few-shot fine-tuning BERT model with
staff query logs
Feature 3: Predictive Maintenance Scheduling
Model Type: LSTM or 1D CNN (based on synthetic time-series sensor data)
Synthetic  Data  Structure: -  Equipment:  Cranes,  Stackers,  Trucks  -  Sensors:  Temperature,  Pressure,
Vibration, Load - Event Timelines: Loading/Unloading timestamps
Output: - Maintenance Risk Score - Predicted Failure Window
Evaluation Metrics: - Precision, Recall, F1-Score (for fault prediction) - Mean Time Before Failure (simulation
validation)
Fine-Tuning  Strategy: -  Model  ensemble  tuning  (comparing  LSTM  vs.  1D  CNN)  -  Sensor  signal
augmentation
Feature 4: Route Optimization + Berth Scheduling
Planned Data Inputs (via API): - Berth occupancy - Ship ETA - Handling time
Model Type: - Reinforcement Learning (RL) Agent or heuristic + rule-based model - Possibly lightweight
GNN if terminal layout supports it
Goal: Minimize vessel wait time and maximize berth throughput
Authentication System
Login Flow: - Facial Recognition (CV-based, stored embedding in DB) - Password fallback login
Model: - FaceNet or Dlib + cosine similarity matching
Database Storage: - Securely hashed facial embeddings + salted password hashes

Frontend Structure (React + TailwindCSS)
This is an interactive web application. Every feature has user-driven controls (filters, uploads, run buttons)
and renders live results (charts, tables, alerts). UI is role-based (Ground Staff, Ops Manager, Maintenance
Lead, Admin).

Assistant - Feature 3: Predictive Maintenance - Feature 4: Berth Scheduling (Phase II) - Admin (only Admin)

Backend Architecture (FastAPI + PostgreSQL + Docker)
Design Principles - Microservices per feature for clear ownership and independent scaling. - API Gateway
for auth, routing, rate limits. - Shared Observability (logs/metrics/traces) across services.
Core API Services (logical) - auth-service: login, facial verification, user roles - forecast-service: Feature 1
training/inference -  anomaly-service: Feature 2A autoencoder detection -  assistant-service: Feature 2B
chatbot + escalation classifier + text2sql - maintenance-service: Feature 3 scoring + scheduling - berth-
service: Feature 4 optimizer (Phase II) - notification-service: email/SMS/in-app alerts, SLAs - data-service:
dataset ingestion, validation, feature store access
Storage -  PostgreSQL:  users/roles,  issues,  alerts,  model  runs,  configs  -  Object  storage  (MinIO/S3-
compatible): model artifacts, uploads, synthetic data batches
Containerization (Docker) - Each service runs in its own Docker container. - Docker Compose (local dev)
and optional lightweight orchestration (free-tier) for deployment.
